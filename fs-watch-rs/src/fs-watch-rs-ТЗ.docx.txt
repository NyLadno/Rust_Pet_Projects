1.1 Описание продукта
fs-watch-rs — это кроссплатформенная CLI-утилита для мониторинга файловой системы в реальном времени. Инструмент наблюдает за указанной директорией, фиксирует все изменения (создание, изменение, удаление файлов), фильтрует события по заданным критериям и доставляет уведомления через настраиваемые каналы вывода.


Аналоги: inotifywait (Linux), fswatch (macOS), watchman (Facebook). fs-watch-rs отличается единым конфиг-файлом, поддержкой webhook-уведомлений и расширяемой архитектурой репортеров.


1.2 Цели проекта
* Бизнес-цель: создать утилиту, пригодную для DevOps/мониторинга продакшн-директорий
* Учебная цель: освоить ключевые концепции Rust на практическом проекте
* Портфолио-цель: продемонстрировать знание системного программирования на Rust работодателям

3. Нефункциональные требования
3.1 Производительность
* Задержка обнаружения события: не более 100 мс от момента изменения
* Потребление памяти: не более 20 МБ в режиме ожидания
* Утилита не должна создавать заметной нагрузки на CPU в idle


3.2 Надёжность
* Корректная обработка ошибок: несуществующая директория — информативное сообщение, не panic
* Корректное завершение по Ctrl+C (graceful shutdown)
* Недоступность webhook не должна прерывать основной мониторинг


3.3 Качество кода
* Без предупреждений компилятора (cargo build --release без warnings)
* Покрытие юнит-тестами: минимум 3 теста для логики парсинга событий
* Документирование публичных структур и трейтов (cargo doc)
* Форматирование: cargo fmt перед каждым коммитом
* Линтер: cargo clippy без ошибок уровня deny

4.2 Ключевые типы и трейты
Трейт Reporter
pub trait Reporter: Send + Sync {
    fn report(&self, event: &WatchEvent) -> Result<(), AppError>;
    fn flush(&self) -> Result<(), AppError> { Ok(()) }
}


Тип WatchEvent
pub struct WatchEvent {
    pub kind:      EventKind,       // Created | Modified | Deleted | Renamed
    pub path:      PathBuf,
    pub timestamp: DateTime<Local>,
}
 
pub enum EventKind {
    Created,
    Modified,
    Deleted,
    Renamed { from: PathBuf },
}


Структура Config
#[derive(Debug, Deserialize)]
pub struct Config {
    pub watch_path:  PathBuf,
    pub recursive:   bool,
    pub filters:     Vec<String>,    // расширения
    pub output_file: Option<PathBuf>,
    pub webhook_url: Option<String>,
    pub debounce_ms: u64,
}

6. CLI-интерфейс
6.1 Синтаксис вызова
fs-watch-rs [OPTIONS] <PATH>
 
Arguments:
  <PATH>  Путь к наблюдаемой директории
 
Options:
  -r, --recursive              Наблюдать рекурсивно [default: false]
  -f, --filter <EXT>           Фильтр по расширению (.rs, .toml, ...)
  -p, --pattern <GLOB>         Фильтр по glob-паттерну (*.log, src_*)
  -o, --output <FILE>          Путь к файлу лога
  -c, --config <FILE>          Путь к config.toml
      --webhook <URL>          URL для POST-уведомлений
      --debounce <MS>          Дебаунс событий в мс [default: 200]
      --no-color               Отключить цветной вывод
  -h, --help                   Показать справку
  -V, --version                Показать версию


6.2 Примеры использования
# Простой мониторинг директории
fs-watch-rs ./src
 
# Рекурсивно, только .rs файлы
fs-watch-rs ./project --recursive --filter .rs
 
# С записью в файл и webhook
fs-watch-rs ./logs -r --output events.log --webhook http://localhost:8080/hook
 
# С конфиг-файлом
fs-watch-rs --config ./config.toml .


6.3 Формат конфиг-файла (config.toml)
watch_path  = "./src"
recursive   = true
debounce_ms = 300
filters     = [".rs", ".toml"]
 
[output]
file        = "./watch.log"
 
[webhook]
url         = "https://hooks.example.com/notify"
timeout_sec = 5

8. План реализации (MVP)
Подход: итеративная разработка. Каждая итерация — рабочий, запускаемый бинарник.


Итерация 1: Скелет проекта и CLI  ⏱ 2–3 дня
* cargo new fs-watch-rs --bin, настройка Cargo.toml с зависимостями
* Структуры Cli через clap (derive): поля path, recursive, filter, output
* main.rs: парсинг аргументов, вывод в stdout для проверки
* Модуль errors.rs с AppError через thiserror
* Что изучишь: derive-макросы, Result<T,E>, базовая структура проекта
	

Итерация 2: Базовый watcher и события  ⏱ 3–4 дня
* Модуль event.rs: тип WatchEvent, enum EventKind с полями
* Модуль watcher.rs: подключение notify, цикл обработки событий через match
* Маппинг событий notify -> WatchEvent (конвертация через From/Into)
* Вывод событий в stdout с временной меткой (chrono)
* Что изучишь: enum + pattern matching, From/Into трейты, работа с внешним крейтом
	

Итерация 3: Фильтрация и трейт Reporter  ⏱ 3–4 дня
* Логика фильтрации: по расширению (--filter) и glob-паттерну (--pattern)
* Трейт Reporter с методом report(&self, event: &WatchEvent) -> Result<()>
* ConsoleReporter: цветной вывод через colored
* FileReporter: запись в файл через BufWriter
* Что изучишь: трейты, trait objects (Box<dyn Reporter>), BufWriter, ownership при передаче в функции
	

Итерация 4: Конфиг-файл  ⏱ 2–3 дня
* Структура Config с #[derive(Deserialize)]
* Загрузка config.toml через toml::from_str
* Слияние: аргументы CLI имеют приоритет над config.toml
* Валидация Config: существует ли директория, корректны ли расширения
* Что изучишь: serde Deserialize, Option<T>, обработка вложенных структур
	

Итерация 5: Webhook (async)  ⏱ 3–4 дня
* WebhookReporter: отправка POST JSON через reqwest + tokio
* #[tokio::main] в main.rs или spawn для webhook в отдельном потоке
* Таймаут webhook-запроса: 5 секунд, ошибка не прерывает мониторинг
* Дебаунсинг: события в течение debounce_ms объединяются в одно
* Что изучишь: async/await, tokio::spawn, Arc для shared state, обработка ошибок в async
	

Итерация 6: Тесты, документация, README  ⏱ 2–3 дня
* Юнит-тесты: фильтрация событий, парсинг расширений, конвертация EventKind
* cargo doc: doc-комментарии /// для публичных типов и трейтов
* README.md: описание, установка, примеры с GIF/скриншотами
* Итоговые проверки: cargo fmt, cargo clippy, cargo test
* Что изучишь: #[cfg(test)], assert_eq!, tempfile для тестов файловой системы
